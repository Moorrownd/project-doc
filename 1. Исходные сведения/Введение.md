# Введение

## 1. Цель документа

Целью данного архитектурного репозитория является описание архитектуры системы P2P Parking Sharing(Сервис совместного использования парковочных мест).  
Репозиторий предназначен для фиксации архитектурных решений, требований и ограничений, а также для обеспечения единого понимания структуры системы всеми участниками проекта.

Документ служит основой для проектирования, разработки, сопровождения и масштабирования системы, а также для анализа архитектурных изменений на протяжении жизненного цикла проекта.

---

## 2. Область применения

Архитектурный репозиторий охватывает систему P2P Parking Sharing, предназначенную для соединения владельцев частных парковочных мест и водителей, нуждающихся в парковке.

В область применения входят:
- веб-приложение и мобильное приложение системы;
- серверная часть, реализованная в виде микросервисной архитектуры;
- процессы поиска, бронирования и оплаты парковочных мест;
- взаимодействие с внешними сервисами (платежные и картографические сервисы, сервисы уведомлений).

Репозиторий не охватывает внутреннюю архитектуру сторонних внешних сервисов и инфраструктурных решений, находящихся вне контура системы.

---

## 3. Целевая аудитория

Данный репозиторий предназначен для следующих групп заинтересованных лиц:
- **Разработчики** — для понимания структуры системы, взаимодействия компонентов и архитектурных ограничений;
- **Архитекторы** — для анализа архитектурных решений и их соответствия требованиям;
- **Менеджеры проекта** — для общего понимания архитектуры и оценки рисков;
- **Операторы системы** — для понимания логики работы основных бизнес-процессов;
- **Тестировщики** — для формирования тестовых сценариев на основе архитектуры и бизнес-процессов.

---

## 4. Основные принципы и стандарты

Архитектура системы строится на следующих принципах:
- микросервисная архитектура с четким разделением ответственности;
- централизованная аутентификация и авторизация пользователей;
- асинхронное взаимодействие между сервисами для повышения надежности и масштабируемости;
- приоритет безопасности и защиты персональных данных пользователей;
- высокая доступность критически важных функций (бронирование и оплата).

Для описания архитектуры используются:
- **C4-диаграммы** (контекстная и контейнерная) для визуализации структуры системы;
- **UML-диаграммы последовательностей** для описания ключевых бизнес-сценариев.

---

## 5. Терминология

В репозитории используются следующие ключевые термины:

- **Владелец парковочного места** — пользователь, предоставляющий парковочное место в аренду.
- **Арендатор (Водитель)** — пользователь, осуществляющий поиск и бронирование парковочного места.
- **Оператор системы** — пользователь, управляющий системой и разрешающий спорные ситуации.
- **Бронирование** — процесс резервирования парковочного места на определенный временной интервал.
- **Платеж** — безналичная операция оплаты бронирования через внешний платежный сервис.
- **Микросервис** — изолированный компонент системы, отвечающий за отдельную бизнес-функцию.
- **API Gateway** — компонент, обеспечивающий единую точку входа для клиентских приложений.

## 6. Структура репозитория

Репозиторий включает следующие основные разделы:
- **Введение** — общее описание целей и области применения репозитория;
- **Архитектурные факторы** — функциональные и нефункциональные требования, ограничения и принципы;
- **Архитектурные диаграммы** — C4 и UML-диаграммы системы;
- **Описание компонентов** — детализация микросервисов и их ответственности;
- **Решения и ограничения** — зафиксированные архитектурные решения и допущения.
